Managing Requirements

From Epics to Executable Stories:
Decomposing, Planning, and Refining Your Backlog

Welcome to Module 3, Managing Requirements. This is the largest module in our course and arguably the most critical. Everything we discussed in Modules 1 and 2 about Agile estimation discipline and why plans fail converges here. Before we can estimate anything, we need to understand WHAT we are estimating. This module covers the full lifecycle of a requirement: from high-level scope decomposition, through release planning and backlog management, all the way down to well-formed, estimable user stories. By the end of this module, participants will be equipped to take a vague product vision and systematically break it into work items that a team can confidently estimate and deliver.

Module Objectives

  • Decompose project scope from epics down through themes, features, user stories, and tasks
  • Construct a release plan that maps stories to iterations and adapts as velocity stabilizes
  • Apply the INVEST criteria to write high-quality, estimable user stories
  • Identify and prevent the "Hidden Waterfall" anti-pattern in sprint planning
  • Manage the product backlog using grooming metrics and the Definition of Ready
  • Select appropriate story scales (Fibonacci, T-shirt) for different estimation contexts
  • Use spikes and "Get Smart" stories to reduce uncertainty before committing to estimates
Walk through these objectives so participants know what to expect. Emphasize that this module bridges the gap between understanding Agile estimation philosophy from Modules 1-2 and actually sizing work which we will cover in Module 4 on Relative Sizing Metrics. Managing requirements well is the prerequisite for accurate estimation. If the stories are poorly written or the backlog is a mess, no estimation technique will save you.

Decomposing Scope

Breaking the big picture into estimable pieces

We begin with scope decomposition. This is the foundational skill for Agile requirements management. In traditional project management, we created a Work Breakdown Structure. In Agile, we have an analogous but more flexible hierarchy: Epics, Themes, Features, User Stories, and Tasks. Let us explore each level and understand why decomposition is essential for estimation accuracy.

📦 The Scope Hierarchy

Walk through the hierarchy from top to bottom. An EPIC is the largest container - think User Account Management or Payment Processing System. THEMES group related stories by a business concern. FEATURES are distinct capabilities a user can interact with. USER STORIES are the primary unit of estimation - small enough for a single sprint. TASKS are the implementation steps, typically estimated in hours. Teams estimate at the story level, not the task level. The right-hand example shows a concrete decomposition of an E-Commerce Checkout epic through all five levels.

  • Exposes hidden complexity early
Key Insight: You cannot accurately estimate what you do not understand. Decomposition forces understanding.

Developing the Release Plan

Mapping stories to iterations for predictable delivery

Now that we understand how to decompose scope, the next question is: how do we organize all these stories into a delivery timeline? That is the purpose of the release plan. The release plan is the bridge between the product backlog and the sprint-level execution. It answers the question: Given our current velocity, when can we expect to deliver this set of features?

🗓️ What a Release Plan Contains

📋 Core Components

  • Release goal — the business outcome targeted
  • Prioritized feature list — what is in scope
  • Estimated velocity — points per iteration
  • Number of iterations — sprint count and cadence
  • Story-to-iteration mapping — which stories go where
  • Key milestones and external dependencies

The release plan is NOT a commitment to a specific set of features on a specific date. It is a forecast based on current knowledge. Emphasize the distinction: fixed date means variable scope, fixed scope means variable date. Agile prefers to fix the date and flex the scope. The table on the right shows a simplified release plan: 13 stories totaling 83 points spread across 4 sprints with an average velocity of 21 points per sprint. In practice, the Product Owner would sequence stories by priority so that if we run out of time, the most valuable stories are already delivered.

🗓️ The Release Planning Process

📝 Ceremony Steps

  1. PO presents the release vision and goals
  2. Team reviews the prioritized backlog
  3. Team provides high-level estimates
  4. Calculate expected velocity (historical or forecasted)
  5. Map stories into iterations by priority order
  6. Identify risks, dependencies, and unknowns

Participants: PO + Scrum Master + Dev Team + Stakeholders
Duration: Half-day (3-mo release) to full day (6-mo) • Revisit every 2-3 sprints

Key Principle: The release plan is a living document. Update every 2-3 sprints as actual velocity replaces forecasts.

📉 Velocity Lower?

  • Cut lower-value stories / negotiate scope
  • Extend timeline if date is flexible
  • Investigate impediments

📈 Velocity Higher?

  • Pull in additional backlog stories
  • Invest in tech debt / quality
  • Validate estimates are not being gamed
Release planning is a collaborative ceremony involving PO, Scrum Master, dev team, and stakeholders. The PO drives priorities while the team provides estimates and identifies risks. After 2-3 sprints, real velocity data replaces forecasts. If velocity is lower than expected, the PO must make scope decisions - this is healthy inspect-and-adapt. If higher, confirm the team is not cutting corners on the Definition of Done. Never inflate velocity by lowering quality standards.

Leveraging Themes

Organizing stories by business value streams

Themes are a powerful organizational tool that is often underutilized by Agile teams. While epics and stories define WHAT we are building, themes help us understand WHY we are building it and how different stories relate to strategic business objectives.

🏷️ Themes — Grouping Stories by Business Value

What Is a Theme?

A collection of related stories supporting a specific business objective. Themes cut across epics, represent strategic goals, and help stakeholders understand investment allocation.

🛡️ Security: 2FA, encryption, PCI compliance
📱 Mobile: responsive design, touch gestures, offline
⚡ Performance: caching, CDN, image optimization
📊 Analytics: event tracking, A/B testing, funnels

INVEST-ing in Good Stories

The six criteria for well-formed user stories

Bill Wake coined the INVEST acronym as a checklist for high-quality user stories. This is arguably the most practical tool in this entire module. A story that satisfies all six INVEST criteria will be dramatically easier to estimate, plan, and deliver.

📖 The INVEST Criteria

 CriterionMeaning✅ Good Example❌ Poor Example
I Independent No dependency on other stories "User can reset password via email" "Build payment API" then "Build payment UI"
N Negotiable Details open for discussion "User can filter products" "Implement exactly 3 dropdowns with hex #2b6cb0"
V Valuable Delivers tangible user/business value "Customer can track order status" "Refactor database schema"
E Estimable Enough info to size it "Display top 10 products by rating" "Integrate with partner API"
S Small Completable in one sprint (1-3 days) "Add email validation to signup form" "Build user management system"
T Testable Clear acceptance criteria "User sees error for invalid card" "Make the app more user-friendly"
Bill Wake coined the INVEST acronym as a checklist for high-quality user stories. Go through each row: Independent means stories can be built in any order. Negotiable means the story is a conversation starter, not a locked specification. Valuable means every story delivers something a user cares about. Estimable means the team understands enough to size it. Small means completable in one sprint. Testable means clear acceptance criteria exist. The poor examples illustrate common violations - forced dependencies, over-specification, no user value, vagueness, excessive scope, and unmeasurable criteria.

📖 The User Story Template

The Standard Format
As a [type of user],
I want [an action or feature],
So that [a benefit or value].

✅ Well-Written Story

As a returning customer,
I want to save my shipping address,
so that I can check out faster on future orders.

Acceptance Criteria:
• User can save up to 5 addresses
• Default address is pre-selected at checkout
• User can edit or delete saved addresses

❌ Poorly-Written Story

As a user,
I want the address feature,
so that it works.

Problems:
• "User" is too vague — which user?
• "Address feature" is ambiguous
• "It works" provides no testable value
• No acceptance criteria

The three-part template - As a, I want, So that - forces the author to think about WHO benefits, WHAT they need, and WHY it matters. The so that clause is often omitted but is the most important part. It connects the story to business value and helps the team understand the intent behind the request. This understanding is crucial for estimation because it helps the team make implementation decisions.

📖 Story Splitting Techniques

When a story is too large, use these patterns to split it:

✂️ Splitting Patterns

  1. By workflow steps
    Search → Filter → Sort → Paginate
  2. By business rules
    Basic validation → Complex validation
  3. By data variations
    Credit card → PayPal → Crypto
  4. By CRUD operations
    Create → Read → Update → Delete
  5. By happy/sad paths
    Success flow → Error handling
⚠️ Avoid These Splits:
  • By architectural layer — "Build the API" / "Build the UI" creates dependencies
  • By technology — "Set up database" / "Write backend" is not user-valuable
  • Arbitrary halves — "Part 1" / "Part 2" is meaningless
💡 Golden Rule: Every split story must deliver a thin vertical slice of user-visible functionality — touching all layers from UI to database.
Story splitting is an essential skill. When a story is estimated at 13 or more points, it usually needs to be split. The five patterns on the left are proven approaches. The anti-patterns on the right are critical. Splitting by architectural layer creates dependencies and delivers no user value until both are complete. This is actually the Hidden Waterfall pattern we will discuss next.

Understanding The Hidden Waterfall

The anti-pattern that undermines Agile delivery

The Hidden Waterfall is one of the most common and damaging anti-patterns in Agile teams. It occurs when a team appears to be doing Agile but the actual work within those sprints follows a waterfall sequence.

💧 Spotting & Fixing the Hidden Waterfall

Sprint 1: Analysis → Sprint 2: Design → Sprint 3: Code → Sprint 4: Test → Sprint 5: Deploy
❌ Waterfall with sprint-length phases!

🚩 Warning Signs

  • No working software until Sprint 3+
  • Sprint reviews with "nothing to demo"
  • Specialists idle in certain sprints
  • Stories split by layer (API / UI / DB)

✅ The Fix: Vertical Slices

Every sprint: Analyze + Design + Code + Test → Done ✓
  • Write vertical stories touching all layers
  • Pair specialists (backend + frontend)
  • Invest in T-shaped skills
  • Swarm: finish one story before starting another
  • Measure working software, not phase completion
🎯 Goal: Every sprint review should include a live demonstration of new, working functionality.
The Hidden Waterfall is one of the most common anti-patterns. The team adopts Agile ceremonies but works in sequential phases. Sprint 1 is all analysis, Sprint 2 is design, etc. No working software is delivered until Sprint 3 at the earliest. The fix is vertical slices: all engineering activities happen within every sprint. Each sprint delivers a complete, working slice of functionality. The swarm technique is effective - the whole team focuses on finishing one story at a time, reducing WIP and increasing flow.

Managing the Product Backlog
and Metrics for Grooming

Keeping the backlog healthy and sprint-ready

A well-maintained product backlog is the foundation of effective sprint planning and accurate estimation. If the backlog is a disorganized list of hundreds of items with no refinement, the team cannot plan or estimate effectively. Backlog grooming, also called backlog refinement, is the ongoing process of keeping the backlog in shape.

📚 Backlog Grooming / Refinement

🔄 The Refinement Ceremony

  • Frequency: Once or twice per sprint (mid-sprint is common)
  • Duration: 5-10% of sprint capacity (e.g., 2-4 hours for a 2-week sprint)
  • Participants: PO + Development Team (+ stakeholders as needed)
  • Scrum Guide: Not a formal event, but an essential ongoing activity

📋 Refinement Activities

  • Clarify story details and acceptance criteria
  • Split stories that are too large
  • Estimate or re-estimate stories
  • Re-prioritize based on new information
  • Remove obsolete stories
  • Add newly identified stories

✅ Definition of Ready (DoR)

A story meets the Definition of Ready when:

  • Acceptance criteria are written and understood
  • Story has been estimated by the team
  • Story is small enough for one sprint
  • Dependencies are identified and resolved
  • The team can explain the story to a stakeholder
Backlog refinement is the ongoing process of preparing stories for upcoming sprints. The Scrum Guide recommends spending about 10% of sprint capacity on refinement. The Definition of Ready is the complement to the Definition of Done. While DoD says when is a story finished, DoR says when is a story ready to be pulled into a sprint. A story that does not meet the DoR should not be brought into sprint planning.

📚 Grooming Metrics & Backlog Health

MetricTargetWhy It Matters

These metrics help the team assess backlog health. Story Readiness Rate targets 80% or higher - the percentage meeting the Definition of Ready before sprint planning. Churn Rate measures stories changed mid-sprint; high churn suggests the PO is not prioritizing effectively. The iceberg visual shows the refinement gradient: top items are sprint-ready, middle items need work, bottom items are raw ideas. The key insight: do not spend hours refining stories that might never be built. Focus energy on the top.

Examining Story Metrics
and the Story Scale

Choosing the right scale for estimation

Now we turn to the scales we use for sizing stories. In Module 4 we will dive deep into relative sizing techniques. Here we introduce the most common scales and when to use each one. The choice of scale affects the precision of your estimates and the speed of the estimation process.

📏 Story Estimation Scales

🔢 Modified Fibonacci

1 2 3 5 8 13 21 40 100
  • Gaps increase with size — reflects uncertainty
  • No "9" or "17" — prevents false precision
  • Stories above 13 must be split

Use for: Sprint planning, release planning

👕 T-Shirt Sizing

XS S M L XL XXL
  • Fast, low-effort estimation
  • Intentionally coarse-grained
  • Can be mapped to point ranges later

Use for: Roadmap, epic sizing, large groups, discovery

⚠️ Avoid: Linear scales (1-10) encourage false precision • Hours-based estimation conflates size with duration • Mixing scales within the same planning horizon
Walk through both scales. Fibonacci is the most common for story points. The gaps increase as numbers get larger, reflecting that large stories have inherently higher uncertainty. The difference between a 1-point and 2-point story is meaningful; the difference between 20 and 21 points is noise. T-shirt sizing is faster, used for roadmap and epic-level planning. Use Fibonacci when you need precision for near-term planning, T-shirts when speed matters more. Never estimate stories in hours - hours conflate size with duration and vary by person.

This reference chart provides a starting point for teams establishing their estimation scale. Emphasize that the chart is team-specific. What matters is internal consistency within a team, not cross-team comparisons. A 5-point story for Team A and a 5-point story for Team B may represent very different amounts of work. The Action column is particularly important. Stories at 8 points should trigger the question can we split this? Stories at 13 must be split. Anything at 21 or above is an epic that needs decomposition.

Using Spikes and
"Get Smart" Stories

Reducing uncertainty through time-boxed research

What happens when the team encounters a story they simply cannot estimate? The answer is usually: we need more information. That is where spikes and Get Smart stories come in. These are deliberate, time-boxed investments in learning that reduce uncertainty and enable better estimation.

🧪 What Is a Spike?

💡 Spike Definition

A spike is a time-boxed research or prototyping task designed to answer a specific question or reduce uncertainty about a story or technical approach.

Origin: The term comes from Extreme Programming (XP). Just as a railroad spike is hammered in to test if the ground is solid, an Agile spike tests if an approach is viable.

⏱️ Key Characteristics

  • Time-boxed: Fixed duration (typically 1-3 days)
  • Question-driven: Must answer a specific question
  • Disposable output: Prototype code is thrown away
  • Knowledge is the deliverable — not production code
💡 Key Insight: The output of a spike is knowledge, not production code. Any prototype code should be discarded and rewritten properly.
Spikes are one of the most valuable but also most misused tools in Agile. A spike is appropriate when the team genuinely cannot estimate because they lack critical information. The key constraint is the time-box. A spike should be 1-3 days maximum. If the question cannot be answered in that time, the question is too broad and needs to be decomposed. The output of a spike is knowledge, not production code. Any prototype code written during a spike should be discarded and rewritten properly.

🧪 When to Use Spikes

✅ When to Use a Spike

  • Team cannot estimate a story — too many unknowns
  • New technology or API that no one has used before
  • Architectural decision with multiple viable options
  • Performance feasibility — "Can we handle 10K txn/s?"
  • Third-party integration complexity is unknown

❌ When NOT to Use a Spike

  • The team is just uncomfortable estimating — push through it
  • The question can be answered by a 15-minute conversation
  • As an excuse to avoid committing to an estimate
  • When the story is simply too large — split it instead
⚠️ Rule of Thumb: If a team is running more than 2 spikes per sprint, the backlog needs more refinement work upstream.
Walk through the two lists carefully. On the left, the legitimate use cases: genuine unknowns that block estimation. On the right, the misuses: discomfort is not the same as inability. If the team can have a short conversation and resolve the uncertainty, a spike is overkill. If the story is simply large, the answer is decomposition, not a spike. The rule of thumb about 2 spikes per sprint is a useful guardrail.

🧪 "Get Smart" Stories & the Spike Feedback Loop

🧠 What Is a "Get Smart" Story?

A spike variant focused on reducing estimation uncertainty for specific upcoming stories.

Format:
As a development team,
We want to investigate [specific question],
So that we can estimate stories [S-14, S-15, S-16].

📋 Examples

  • "Get Smart on Payment Gateway" — compare Stripe vs Braintree
  • "Get Smart on Real-Time Updates" — prototype WebSocket vs SSE
Unestimable Story ❓ ▼ Spike / Get Smart 🔍 ▼ Knowledge Gained 📋 ▼ Estimable Stories ✅
📊 Tracking Spikes:
  • Estimated in hours, not story points
  • Do not count toward velocity
  • >2 spikes/sprint = backlog needs more refinement
  • Measure: did the team estimate successfully afterward?
Get Smart stories are a spike variant explicitly tied to specific backlog items. The output is the ability to estimate previously unestimable stories. The feedback loop is straightforward: unestimable story triggers a spike, the spike produces knowledge, knowledge enables confident estimation. For tracking: spikes are time-boxed and estimated in hours, not story points. They do not count toward velocity. If a team does 3 or more spikes per sprint, the backlog is not being refined far enough ahead.

Lab 3: Story Decomposition and INVEST Workshop

Hands-on practice applying the concepts from this module

  • 📦 Part A: Decompose an epic into themes, features, and stories
  • 📖 Part B: Evaluate stories against INVEST and improve them
  • ✂️ Part C: Split large stories using vertical slice patterns
  • 🧪 Part D: Write "Get Smart" stories for uncertain items

This is a substantial hands-on lab that reinforces all the key concepts from this module. Part A exercises scope decomposition. 🎯 Key Takeaways

  • 📦 Decompose relentlessly: Smaller items yield more accurate estimates and expose hidden complexity.
  • 🗓️ Release plans are living documents: Update every 2-3 sprints; fix the date, flex the scope.
  • 🏷️ Themes connect work to strategy: Track capacity allocation across business value streams.
  • 📖 INVEST in every story: Independent, Negotiable, Valuable, Estimable, Small, Testable.
  • 💧 Avoid the Hidden Waterfall: Deliver working software each sprint via vertical slices.
  • 📚 Groom the iceberg: Keep the backlog top refined; use the Definition of Ready.
  • 📏 Choose scales wisely: Fibonacci for precision, T-shirt sizing for speed.
  • 🧪 Spike before you guess: Time-boxed research is cheaper than wrong estimates.
